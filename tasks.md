# Project LatteCam MVP 任务拆分清单

本文档基于 `prd.md` 拆分开发任务，用于进入编码前审核范围、顺序、依赖与验收边界。当前阶段只做任务规划，不创建 Xcode 工程、不编写 Swift 代码。

## 0. 总体实施原则

### 0.1 MVP 目标边界

- 目标：旧 iPhone 作为前台常亮的音视频采集端，通过带认证的 RTMPS 推流到 MacBook Pro。
- 目标：MacBook Pro 运行 `MediaMTX`，接收 RTMPS 并仅向本机 Scrypted 输出 localhost RTSP。
- 目标：Scrypted 从 `MediaMTX` 拉取 RTSP，并桥接到 HomeKit / HKSV。
- 不做：iPhone 本机 RTSP Server。
- 不做：锁屏或后台摄像头采集。
- 不做：本地录像、AI 检测、滤镜、多镜头切换。

### 0.2 推荐开发顺序

1. 先打通 MacBook Pro 上的媒体中转链路。
2. 再创建 iOS SwiftUI 工程骨架。
3. 再接入 HaishinKit，实现最小 RTMPS 推流。
4. 再补 UI 状态、自动重连、节能模式。
5. 最后接入 Scrypted 与 HomeKit。

### 0.3 审核节奏

- 每完成一个一级任务，需要对照对应验收点确认后再进入下一阶段。
- P0 任务先完成可用闭环，P1 任务在闭环稳定后再做。
- 任何会改变架构方向的技术发现，需要先更新 `prd.md` 和本文档，再继续实现。

## 1. MacBook Pro 本地媒体中转准备

### 1.1 安装与运行 MediaMTX

- 1.1.1 确认 MacBook Pro 系统环境和芯片架构。
- 1.1.2 选择 `MediaMTX` 安装方式。
- 1.1.3 启动 `MediaMTX` 本地服务。
- 1.1.4 确认 RTMPS 监听端口可用，默认 `1936`。
- 1.1.5 确认 RTSP 输出端口仅在 `127.0.0.1:8554` 可用。

验收点：

- MacBook Pro 上 `MediaMTX` 能正常启动。
- 局域网内其他设备可以访问 MacBook Pro 主机名或 IP。
- `1936` 和 `127.0.0.1:8554` 没有被其他服务占用。

### 1.2 配置基础流路径

- 1.2.1 确定 MVP 流名称为 `live/lattecam`。
- 1.2.2 确定 iPhone RTMPS 推流地址格式：`rtmps://YOUR_MAC_HOST.local:1936/live/lattecam`。
- 1.2.3 确定 Scrypted RTSP 拉流地址格式：`rtsp://127.0.0.1:8554/live/lattecam`。
- 1.2.4 如主机名解析不稳定，准备 IP 地址版本作为备用。

验收点：

- 文档中明确一组主地址和一组备用 IP 地址格式。
- iPhone App 后续只需要配置 RTMPS 推流目标与发布认证。
- Scrypted 后续只需要配置 RTSP 拉流地址。

### 1.3 MacBook Pro 长时间运行设置

- 1.3.1 设置 MacBook Pro 接电运行。
- 1.3.2 防止系统睡眠中断服务。
- 1.3.3 检查 macOS 防火墙对 RTMPS / Scrypted / HomeKit 端口的影响。
- 1.3.4 记录重启服务的手动流程。

验收点：

- MacBook Pro 合盖、睡眠、防火墙策略不会意外中断 MVP 测试。
- 出现服务异常时，可以手动恢复 `MediaMTX`。

## 2. iOS 工程创建与基础配置

### 2.1 创建 Xcode 项目

- 2.1.1 在当前目录创建新的 iOS App 工程，项目名建议为 `LatteCam`。
- 2.1.2 使用 SwiftUI 生命周期。
- 2.1.3 语言使用 Swift。
- 2.1.4 最低 iOS 版本根据旧 iPhone 真实系统版本确定，初始建议不低于 HaishinKit 当前要求。
- 2.1.5 确认项目可以在 Xcode 中打开和编译。

验收点：

- 当前目录中出现 `LatteCam` 工程。
- 工程可以在真机目标上完成一次空项目构建。

### 2.2 权限与系统能力配置

- 2.2.1 添加相机权限描述 `NSCameraUsageDescription`。
- 2.2.2 添加麦克风权限描述 `NSMicrophoneUsageDescription`。
- 2.2.3 配置 App Transport Security，允许本地网络访问并使用受信任本地 CA 支持 RTMPS。
- 2.2.4 确认不申请后台摄像头能力，避免误导实现目标。

验收点：

- 首次启动会触发相机与麦克风权限请求。
- 权限说明文案清晰解释用途。
- 未承诺或配置不可实现的后台摄像头能力。

### 2.3 依赖接入规划

- 2.3.1 通过 Swift Package Manager 添加 `HaishinKit.swift`。
- 2.3.2 确定使用 RTMP/RTMPS 相关模块。
- 2.3.3 记录依赖版本与最低 Xcode / Swift 要求。
- 2.3.4 准备依赖接入失败时的回退处理方式。

验收点：

- 工程能够解析并构建 HaishinKit 依赖。
- 依赖模块选择和 PRD 中的 RTMPS MVP 保持一致。

## 3. iOS 采集与推流核心

### 3.1 StreamManager 状态模型

- 3.1.1 定义推流状态：未配置、权限缺失、停止、连接中、推流中、重连中、失败。
- 3.1.2 定义用户可见错误信息。
- 3.1.3 定义当前 RTMPS 推流目标。
- 3.1.4 定义桥接 RTSP 地址展示字段。
- 3.1.5 确定状态变化对 UI 的通知方式。

验收点：

- UI 能根据状态准确展示当前推流阶段。
- 错误状态不会只停留在日志中，用户能看懂大致原因。

### 3.2 相机与麦克风采集

- 3.2.1 请求并检查相机权限。
- 3.2.2 请求并检查麦克风权限。
- 3.2.3 默认选择后置广角摄像头。
- 3.2.4 默认选择机身麦克风。
- 3.2.5 将采集源绑定到 HaishinKit 的媒体混合/推流对象。

验收点：

- 真机上能看到实时预览。
- 权限拒绝时不会崩溃，并提示用户授权。
- 默认使用后置摄像头和麦克风。

### 3.3 编码参数配置

- 3.3.1 配置视频编码为 H.264。
- 3.3.2 配置 MVP 默认分辨率为 720p。
- 3.3.3 配置帧率为 30fps。
- 3.3.4 配置 GOP 为 1-2 秒。
- 3.3.5 配置音频为 AAC。
- 3.3.6 配置音频码率为 64kbps 或 128kbps。

验收点：

- VLC 或 Scrypted 能识别音视频编码。
- 画面延迟和发热在可接受范围内。
- 如旧 iPhone 压力过高，可以降低码率或分辨率。

### 3.4 RTMPS 推流控制

- 3.4.1 实现启动推流流程。
- 3.4.2 实现停止推流流程。
- 3.4.3 连接 `rtmps://YOUR_MAC_HOST.local:1936/live`。
- 3.4.4 发布流名 `lattecam?user=<publish-user>&pass=<publish-password>` 或与 MediaMTX 路径一致的认证流名。
- 3.4.5 处理连接成功、发布成功、连接失败、发布失败事件。

验收点：

- 点击启动后，`MediaMTX` 能收到 iPhone 推流。
- 点击停止后，推流连接释放。
- 失败状态可见且可重试。

### 3.5 自动重连机制

- 3.5.1 识别网络断开、MacBook Pro 不可达、RTMP 连接失败等场景。
- 3.5.2 设计重试间隔，例如 2 秒、5 秒、10 秒递增。
- 3.5.3 设定最大重试间隔，避免过度频繁重连。
- 3.5.4 用户手动停止后不得自动重连。
- 3.5.5 重连中状态需要在 UI 上清晰显示。

验收点：

- 短暂断网或重启 `MediaMTX` 后，App 能自动恢复推流。
- 用户主动停止时不会被自动重连逻辑重新启动。

## 6. 端到端验收

### 6.1 RTMPS 接收验收

- 6.1.1 启动 `MediaMTX`。
- 6.1.2 iPhone App 启动带认证的 RTMPS 推流。
- 6.1.3 在 `MediaMTX` 日志中确认流上线。
- 6.1.4 停止推流后确认流下线。

验收点：

- `MediaMTX` 能稳定接收来自 iPhone 的 RTMPS 流。
- 启停状态与服务端日志一致。

### 6.2 本机 RTSP 播放验收

- 6.2.1 在运行 MediaMTX 的 Mac 本机使用 VLC 或 ffprobe 打开 `rtsp://127.0.0.1:8554/live/lattecam`。
- 6.2.2 确认画面正常。
- 6.2.3 确认声音正常。
- 6.2.4 观察首帧时间和播放延迟。
- 6.2.5 观察断线重连后的 Scrypted/VLC 恢复情况。

验收点：

- 本机 Scrypted、VLC 或 ffprobe 可以播放画面和声音。
- 延迟符合家庭监控使用预期。
- iPhone 发热没有导致明显降频或断流。

### 6.3 Scrypted 接入验收

- 6.3.1 在 Scrypted 中添加 RTSP 摄像头。
- 6.3.2 填入 `rtsp://127.0.0.1:8554/live/lattecam`。
- 6.3.3 确认 Scrypted 预览可播放。
- 6.3.4 确认 Scrypted 转发到 HomeKit。

验收点：

- Scrypted 能稳定拉取 RTSP。
- HomeKit 中能看到实时画面。

### 6.4 HomeKit / HKSV 验收

- 6.4.1 在家庭 App 中查看实时画面。
- 6.4.2 在局域网 Wi-Fi 下验证播放。
- 6.4.3 切换到蜂窝网络验证远程播放。
- 6.4.4 验证 HKSV 录像是否按预期由 HomeKit 处理。

验收点：

- 家庭 App 可查看实时监控。
- 蜂窝网络下能通过 Home Hub 远程访问。
- 录像职责不落到 iPhone App。
